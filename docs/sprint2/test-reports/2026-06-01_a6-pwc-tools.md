## ⚠️ 3次/4次语义裁决（重要，供 Maria 复核）

**问题**：dev-plan CP-A6-3/CP-A6-4 文字写"**3 次** 429 / timeout 后抛 TransientError"，但实现 `_http_get_with_retry` 是"首次请求 + 3 次重试 = 共 **4 次** attempt"才抛：

```python
_RETRY_BACKOFF_SECONDS = (1.0, 2.0, 4.0)            # 3 个退避点 = 3 次重试
total_attempts = len(_RETRY_BACKOFF_SECONDS) + 1 = 4  # 首次 + 3 次重试
```

**裁决：实现正确，符合 dev-plan §573 表格的权威语义，非 BUG。** 依据：
1. §573 表格明确写"**超过 3 次重试** → 抛 TransientError"，退避序列 1s/2s/4s 正好是"3 次重试"。首次请求按行业惯例不计入"重试"计数（retry = 失败后的再次尝试，首次尝试不是 retry）。
2. CP-A6-3/CP-A6-4 的"3 次 429"是对"3 次重试"的口语化简写，与表格语义一致，并非要求"第 3 次 429 即抛"。
3. 因此测试按"共 4 次 attempt"断言：
   - `test_cp_a6_3_four_429_raises_transient_error`：返回 4 次 429 触发 TransientError，且 `m_get.call_count == 4`。
   - `test_cp_a6_3b_three_429_then_success_does_not_raise`：返回 3 次 429 + 第 4 次 200 **不抛**（反证第 4 次仍会重试，非 3 次即抛），call_count==4。
   - CP-A6-4 同理用 4 次 timeout 断言。

**建议**：Maria 复核后可顺手把 dev-plan CP-A6-3/CP-A6-4 文字补一句"（首次 + 3 次重试，共 4 次 attempt）"消歧；表格（权威）无需改动。
