1. 任务背景与目标

A1 是 Sprint 2 阶段 A（状态与工具层 P0 基础设施）的入口任务，承担 "数据契约层升级" 角色——所有 sp2 后续节点（resource_scout / planning）、新工具（git_tools / pwc_tools）、新 UI（Streamlit 三页面）都要消费 A1 产出的 TypedDict 字段与配置形态。

A1 解决三个核心需求：

  • 多模型配置（PRD §5.4 / Q-S2-01 RESOLVED）：4 个节点可各自配不同 LLM；引入 LLMConfigSet = default + overrides: Dict[NodeName, LLMConfig]，节点级路由由 A2 helper + A3 单行 diff 接入。
  • 中英双语字段（PRD §4.7.3 / R-S2-05）：UI 展示用中文（title_zh / abstract_zh / tldr_zh），代码生成检索用英文（method_summary_en / hardware_requirements_en），事实层字段（datasets / metrics）保持英文不译。
  • planning revise 透明化（PRD §2.3 / Q-S2-03 RESOLVED）：_planning_revise_count + _planning_user_feedback 记录用户修改次数，UI 在 ≥ 5 次时软提示切 code_only 但不锁按钮（硬上限语义已废弃）。

3. 实现要点

3.1 多模型配置形态

class LLMConfigSet(TypedDict):
    """多模型 LLM 配置集合（架构 §2.1.1.bis）。"""
    default: LLMConfig                      # 全局默认，必填
    overrides: Dict[str, LLMConfig]         # 节点级覆写，允许空

NodeName = Literal["paper_intake", "paper_analysis", "resource_scout", "planning"]

四个节点对应 PRD §2.4 / AC-S2-11 的覆写白名单。overrides 为空 dict 等同 sp1 单模型形态，向后兼容。

3.2 新老形态双兼容兜底

create_initial_state 是 sp1 测试基线零退化的关键路径：

  • 老形态（sp1 调用）：create_initial_state(arxiv_id, <LLMConfig dict 含 base_url>) → 自动包装为 {"default": cfg, "overrides": {}}
  • 新形态（sp2 调用）：create_initial_state(arxiv_id, <LLMConfigSet dict 含 default 子配置>) → 直接透传 + 规整 overrides 字段
  • 非法形态：抛 ValueError（既不含 base_url 也不含 default）

3.3 双语字段语义反转（PRD §4.7.3）

字段sp1 语义sp2 起语义消费方

4. 关键设计决策

4.1 保留 llm_config 镜像字段（与 dev-plan 偏差但合理）

决策：dev-plan L308 原意是删除 llm_config 字段，当前实现保留作为过渡期向后兼容镜像（值始终 = llm_config_set["default"]）。

理由：sp1 既有代码在 3 个位置直读 state["llm_config"]：

  • core/react_base.py:825（B4 ReAct 基础设施，sp2 A3 单行 diff 才会改）
  • tests/test_sprint1_smoke.py:229（sp1 烟雾测试断言）
  • tests/test_graph_e2e.py:254（sp1 e2e 测试断言）

若在 A1 直接删字段，会立刻打破 sp1 168/168 测试基线（CP-A1-9 硬约束）。保留镜像让两条路径并存到 A3 完成后，再做一次 atomic 清理。

4.2 方案 A：A1 顺手补 schema 同步 diff

A1 给 PaperAnalysis TypedDict 加了 2 个新字段后，tests/test_paper_analysis.py::CP1 的字段对齐断言（PaperAnalysis.__annotations__ == PAPER_ANALYSIS_SCHEMA.properties）会立刻挂——这是 dev-plan 把"加 TypedDict 字段"放 A1、"改 schema 同步"放 B1 造成的逻辑耦合。

Maria 决策方案 A：A1 顺手做最小 schema diff（5 行 properties 追加），不动 B1 的 _LANGUAGE_POLICY_SECTION / backfill / Prompt Cache 回归（这些是 B1 真正的工作量）。理由：schema 与 TypedDict 是同一层契约的两个面，本就应原子改动。

8. 给 B1 的实施建议（验收报告 §6 摘要）

  1. 守 R-PC4 字节级幂等：_LANGUAGE_POLICY_SECTION 必须是模块级常量（与 _ANALYSIS_SYSTEM_PROMPT_BODY 同源），插入位置在 SystemMessage 主体之后、动态上下文之前，禁止包含任何运行期变量。
  2. 统一 LLM 配置读取路径：B1 节点改造时一律走 llm_config_set["default"]，不要新增 state["llm_config"] 直读位点（C1 条件约束）。
  3. Prompt Cache 命中率回归：对照 S-3 基线 Rbaseline = 0.7669，AC-S2-08 守门 Rafter ≥ 0.7286（×0.95）。建议 B1 完成后立刻跑 E2 回归。
  4. backfill 兜底沿用 BUG-S1-03 范式：LLM 漏写 _zh / _en 字段时回退为对应主字段值并标记 degraded_nodes。
  5. HumanMessage 通道扩展：输出语言策略段落放 HumanMessage 而非 SystemMessage 主体（架构 §4.5）。
  6. 3 次连跑回归：B1 完成后沿用稳定性治理范式，至少 3 次 e2e 连跑 + 全量回归全绿才能交付测试工程师。
