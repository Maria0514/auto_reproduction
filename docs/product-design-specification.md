## 1. 产品概述

### 1.1 产品定位

本产品是一个基于 LangGraph 的 agentic workflow 论文自动复现系统，面向计算机科学（CS）领域的科研工作者。系统以静态流水线（7 节点 DAG）为骨架、节点内由 ReAct agent 自主调用工具完成各阶段任务，并在编码执行阶段由 coding 与 execution 两个松耦合 agent 经 state 结构化反馈构成修复循环协作（非共享对话历史的 multi-agent 子图）。系统接收一篇论文作为输入，自动完成论文分析、资源搜集、复现规划、编码执行、结果验证和报告生成的全流程，最大限度降低实验复现的人力成本和技术门槛。

### 1.2 目标用户

- **核心用户群**：CS 学科的在校学生（本科高年级、硕士、博士）
- **次要用户群**：高校实验室中的科研工作者
- **用户共性**：具备基础编程能力，有阅读英文论文的习惯，拥有本地机器或实验室服务器的使用权限

### 1.3 核心价值主张

| 价值维度 | 具体描述 |
|---------|---------|
| 效率提升 | 将数天甚至数周的手动复现过程压缩为数小时的自动化流程 |
| 门槛降低 | 用户无需深入理解每一个实现细节即可获得可运行的复现代码 |
| 结果可信 | 自动对比复现结果与论文报告，分析差异原因，提供可靠的结论 |
| 过程透明 | 完整的复现规划供用户审核和调整，而非黑盒操作 |
| 可交互推进 | 编码与执行阶段的 agent 在缺失关键信息（凭证、参数、决策等）时，能主动暂停流程向用户提问，拿到答复后继续，而非空转或直接失败 |

## 2. 用户画像与使用场景

### 2.1 典型用户画像

**用户 A -- 研二学生小李**
- 正在调研"视觉 Transformer"方向，需要复现 3 篇近期论文的实验结果作为基线对比
- 有一台配备单张 RTX 3090 的实验室工作站
- 熟悉 Python 和 PyTorch，但对某些论文使用的框架（如 JAX）不熟悉
- 希望快速获得能跑通的代码，并理解结果是否可靠

**用户 B -- 博一学生小王**
- 阅读了一篇新发布的论文，作者未公开代码
- 需要从零复现核心实验以验证论文声明
- 拥有实验室的云服务器账号（多 GPU）
- 希望系统自动生成代码，他负责审核和微调

**用户 C -- 本科毕业设计学生小张**
- 选题要求复现一篇经典论文
- 本地只有一台笔记本（无 GPU）
- 希望系统至少帮他生成完整的复现代码，即使暂时无法在本地运行
- 后续找到 GPU 资源后再执行

### 2.2 典型使用场景

**场景 1：基于已有仓库的快速复现**
1. 用户输入 arXiv ID（如 `2409.05591`）
2. 系统分析论文，找到关联的 GitHub 仓库
3. 系统评估多个仓库质量后选择最优仓库
4. 系统生成适配用户环境的复现计划
5. 用户审核计划，确认或调整
6. 系统自动克隆仓库、配置环境、执行实验
7. 系统输出结果报告，对比论文数据

**场景 2：无代码论文的从零复现**
1. 用户输入论文标识
2. 系统深度分析论文方法、公式、实验设置
3. 系统查阅相关参考文献获取补充信息
4. 系统生成从零编写的复现代码和完整计划
5. 用户审核代码和计划
6. 系统执行实验并验证结果

**场景 3：只编码不复现**
1. 用户无法提供运行环境（无 GPU、无服务器）
2. 系统完成论文分析和代码生成
3. 系统将完整的可复现代码包和执行说明交付给用户
4. 用户获得 GPU 资源后可自行执行
5. 交付标准采用"最低基准线 + agent 草拟 + 用户审核"机制（详见 §4.4.3）

## 3. 系统功能架构 -- 七步流程

系统的核心工作流分为七个步骤，每个步骤对应明确的输入、输出和交互点。

### 3.1 流程总览

```
步骤1: 论文输入与解析
       |
步骤2: 深度论文分析
       |
步骤3: 资源搜集与评估
       |
步骤4: 复现规划与用户审核  <-- 关键交互点
       |
步骤5: 编码与环境搭建
       |
步骤6: 执行与测试验证
       |
步骤7: 报告生成
```

### 3.2 各步骤详细定义

#### 步骤 1：论文输入与解析

| 属性 | 内容 |
|------|------|
| **输入** | arXiv ID、论文标题、或关键词搜索 |
| **输出** | 论文基础元数据（标题、摘要、作者、关联 GitHub URL、引用数等） |
| **关键交互点** | 论文搜索与确认交互在 UI 层（论文输入页）完成，用户确认目标论文后，graph 接收的是用户已确认的 arXiv ID |
| **建议 Agent 角色** | Paper Intake Agent -- 接收已确认的 arXiv ID，调用 deepxiv_sdk 获取论文元数据（标题、摘要、作者、关联 GitHub URL 等） |

#### 步骤 2：深度论文分析

| 属性 | 内容 |
|------|------|
| **输入** | 步骤 1 确认的论文 |
| **输出** | 结构化的论文分析结果：方法描述、关键公式、实验设置（数据集、评估指标、超参数）、硬件要求 |
| **关键交互点** | 无（系统自动完成，结果在步骤 4 呈现给用户） |
| **建议 Agent 角色** | Paper Analysis Agent -- 逐章节渐进式阅读论文，提取与复现相关的关键信息，必要时查阅参考文献 |

#### 步骤 3：资源搜集与评估

| 属性 | 内容 |
|------|------|
| **输入** | 论文分析结果 + 论文关联信息 |
| **输出** | 可用资源清单（代码仓库、预训练模型、数据集）及推荐选择 |
| **关键交互点** | resource_scout 自动完成仓库搜索与评分排序，不涉及用户交互；候选仓库列表（含质量评分）随复现计划一并在步骤 4（复现规划与用户审核）中展示，用户在审核计划时可查看仓库选择并要求更换 |
| **建议 Agent 角色** | Resource Scout Agent -- 通过 deepxiv github_url、Papers With Code API、web search 搜索关联仓库，通过 git clone 拉取代码并评估仓库质量（官方性、完整性、最近提交时间），自动推荐最优选择 |

#### 步骤 4：复现规划与用户审核

| 属性 | 内容 |
|------|------|
| **输入** | 论文分析结果 + 资源清单 + 用户环境信息 |
| **输出** | 完整的复现计划文档（包括环境配置、执行步骤、预期耗时、资源需求） |
| **关键交互点** | **核心审核点** -- 用户必须确认计划可行性，可调整任何步骤 |
| **建议 Agent 角色** | Planning Agent -- 综合所有信息生成复现计划，接收用户反馈后修订 |

#### 步骤 5：编码与环境搭建

| 属性 | 内容 |
|------|------|
| **输入** | 用户确认的复现计划 |
| **输出** | 可执行的代码和配置好的运行环境 |
| **关键交互点** | 环境配置失败时需告知用户，并提供替代方案；执行过程中缺失关键信息（凭证、参数、决策等）时，agent 主动暂停并向用户提问，拿到答复后继续 |
| **建议 Agent 角色** | Coding Agent -- 有仓库时进行适配（修改配置、补全脚本），无仓库时从零生成复现代码；能"写→跑→看→改"自成闭环（对代码做轻量自验，如导入是否通过、入口脚本能否启动），缺信息时主动向用户提问 |

#### 步骤 6：执行与测试验证

| 属性 | 内容 |
|------|------|
| **输入** | 步骤 5 生成的代码与环境 |
| **输出** | 实验运行结果（指标数值、日志、可视化） |
| **关键交互点** | 运行出错时系统自动进入 execution↔coding 修复循环（最多 10 轮），10 轮失败后暂停由用户选择后续方案（已决策，见 §4.5.3）；执行过程中缺失关键信息（凭证、数据路径、决策等）时，agent 主动暂停并向用户提问，拿到答复后继续 |
| **建议 Agent 角色** | Execution Agent -- 自主编排并执行实验、自适应处理环境差异、监控运行状态、收集结果数据；缺信息时主动向用户提问 |

#### 步骤 7：报告生成

| 属性 | 内容 |
|------|------|
| **输入** | 复现结果 + 论文原始数据 |
| **输出** | 结构化的复现报告 |
| **关键交互点** | 用户可查阅报告并提出追问 |
| **建议 Agent 角色** | Report Agent -- 对比复现结果与论文数据，分析差异原因，生成报告 |

## 4. 核心功能详述

### 4.1 论文输入与解析

#### 4.1.1 基于 deepxiv_sdk 的输入能力

本系统采用 deepxiv_sdk（v0.2.5，2026 年 4 月发布）作为论文输入与解析的基础工具。

**支持的输入方式**：

| 输入方式 | 说明 | 对应 SDK 调用 |
|---------|------|--------------|
| arXiv ID 直接输入 | 用户输入已知的 arXiv ID | `reader.brief(arxiv_id)` |
| 关键词搜索 | 用户通过关键词查找论文 | `reader.search(query, size=N)` |
| 标题搜索 | 用户输入论文标题进行精确匹配 | `reader.search(title, use_fine_rerank=True)` |

**渐进式阅读策略**：

系统利用 deepxiv_sdk 的渐进式读取能力，按需获取论文内容，优化 token 使用效率：

1. **brief** -- 获取论文快速摘要（标题、TLDR、关键词、引用数、GitHub URL），用于初步判断
2. **head** -- 获取论文元数据和章节结构概览，用于规划阅读路径
3. **section** -- 按需读取特定章节（如 Method、Experiments），用于深度分析
4. **raw** -- 在必要时获取完整论文内容

**元数据获取**：

- 通过 `brief` 获取论文关联的 GitHub URL
- 通过 Semantic Scholar 元数据查询获取引用关系和参考文献信息
- 搜索结果包含结构化字段：`arxiv_id`、`title`、`score`、`abstract`、`tldr`、`authors`、`citation_count`、`categories`

#### 4.1.2 论文确认流程

> 以下确认流程在 Streamlit UI 层（论文输入页）完成，不在后端 graph 内部执行。用户在 UI 上完成搜索、浏览、确认后，系统将已确认的 arXiv ID 传入 graph 启动后续流程。

1. 用户在论文输入页输入标识后，UI 层调用 `reader.brief()` 获取论文摘要信息
2. UI 层向用户展示论文标题、摘要、作者等核心信息
3. 用户在 UI 上确认是否为目标论文
4. 若通过搜索输入且结果多于一篇，用户在 UI 上从候选列表中选择

#### 4.1.3 学科范围约束

系统仅处理 CS 领域论文。当用户输入的论文不属于 CS 分类（可通过 `categories` 字段判断，前缀应为 `cs.*`）时，系统应给予提示并建议用户确认。

### 4.2 资源抓取

#### 4.2.1 资源来源与优先级

| 来源 | 搜索目标 | 优先级 |
|------|---------|--------|
| deepxiv github_url | 通过 `brief()` 的 `github_url` 字段获取论文关联仓库链接，通过 `git clone` 拉取代码。MVP 阶段不直接调用 GitHub Search API | 最高 -- 优先使用论文作者的官方仓库 |
| Papers With Code | 通过论文标题或 arXiv ID 查询关联代码仓库 | 高 -- 作为 deepxiv github_url 的补充 |
| Web Search | 通过搜索引擎查找论文相关代码仓库 | 中 -- 前两者未命中时的兜底搜索 |
| HuggingFace / Kaggle | 预训练模型、数据集 | 数据集/模型 -- 获取可复用的模型权重和数据 |

#### 4.2.2 仓库质量评估

当存在多个关联仓库时，resource_scout 自动评估并选择最优仓库，整个资源抓取阶段不涉及用户交互。MVP 阶段的评估基于 `git clone` 后的本地检查，不依赖 GitHub API。评估维度包括：

- **官方性**：是否为论文作者维护的仓库（权重最高，可通过仓库 URL 中的用户名与论文作者匹配判断）
- **完整性**：是否包含完整的训练/评估脚本、文档说明（README）、环境配置文件（requirements.txt 等）
- **活跃度**：最近提交时间（通过 `git log` 获取）
- **兼容性**：与用户环境的兼容程度（框架版本、硬件要求）

> **MVP 不做**：Star 数、Fork 数、issue 响应情况等依赖 GitHub API 的指标。这些指标作为后续版本的增强项（见 §9.2）。

resource_scout 完成评估后，将候选仓库列表（含各维度评分及推荐理由）写入状态，由步骤 4（复现规划与用户审核）统一展示给用户。用户在审核复现计划时可同时查看仓库选择，并在必要时要求更换仓库。

#### 4.2.3 资源抓取的入口

MVP 阶段的搜索路径（按优先级）：

1. **deepxiv github_url**：通过 `reader.brief(arxiv_id)` 获取论文关联的 GitHub URL，作为首选搜索入口
2. **Papers With Code API**：通过论文标题或 arXiv ID 查询关联代码仓库
3. **Web Search**：当前两者未命中时，通过搜索引擎查找论文相关代码仓库

仓库获取统一通过 `git clone` 完成，MVP 阶段不直接调用 GitHub Search API。

- HuggingFace Hub API 可搜索关联模型和数据集（MVP 阶段暂不集成，见 §7.2）

#### 4.2.4 无资源情况处理

当所有来源均未找到可用代码仓库时，系统进入"从零编码"模式（见 4.4.2）。

### 4.3 复现规划与用户审核

#### 4.3.1 复现计划内容

系统生成的复现计划应包含以下结构化内容：

```
复现计划
├── 1. 论文核心信息摘要
│   ├── 论文标题与来源
│   ├── 核心方法概述
│   └── 目标实验列表（对应论文中的哪些表/图）
├── 2. 环境需求
│   ├── 硬件要求（GPU型号/数量、内存、磁盘空间）
│   ├── 软件依赖（框架版本、关键库）
│   └── 预估运行时间
├── 3. 数据准备
│   ├── 所需数据集列表
│   ├── 数据获取方式
│   └── 数据预处理步骤
├── 4. 代码方案
│   ├── 使用的仓库/从零编码声明
│   ├── 关键修改点（如有仓库）
│   └── 核心模块设计（如从零编码）
├── 5. 执行步骤
│   ├── 按顺序排列的执行命令/脚本
│   ├── 每步的预期输出
│   └── 关键检查点
└── 6. 预期结果
    ├── 论文报告的指标值
    └── 可接受的偏差范围（如有经验参考）
```

#### 4.3.2 用户审核交互

1. 系统将完整计划展示给用户，同时展示 resource_scout 提供的候选仓库列表（含仓库名、质量评分、选择理由），当前选用的仓库以高亮标识
2. 用户可以：
   - **确认**：同意计划（包括仓库选择），进入执行阶段
   - **调整**：修改任何部分（如缩减实验范围、更换数据集、调整超参数）
   - **更换仓库**：从候选仓库列表中选择其他仓库，planning 节点基于新的仓库信息重新生成复现计划
   - **追问**：对计划中的某个环节提出疑问
3. 系统根据用户反馈修订计划，直至用户确认。若用户要求更换仓库，planning 节点将基于新仓库重新生成计划并再次提交用户审核

#### 4.3.3 平台感知的规划

系统在生成计划时需考虑用户的运行环境（见第 6 章），自动适配：

- 本机单 GPU 时：调整 batch size、建议梯度累积
- 多 GPU 服务器时：配置分布式训练脚本
- 无运行环境时：生成"只编码"的交付方案

### 4.4 编码与复现执行

#### 4.4.0 编码与执行的 agent 能力形态

> 决策日期：2026-07-01｜状态：**已决策（Sprint 4，路线丙）**｜详见 `docs/sprint4/prd.md`

编码（coding）与执行（execution）是两个松耦合、各自具备自主决策能力的 agent，通过结构化反馈相互协作，7 步流程骨架不变：

- **coding agent（可自验的编码闭环）**：不再只是"盲写"代码后交给下游执行，而是能在编码阶段对代码做轻量自验（如导入是否通过、入口脚本能否启动、有无低级语法/依赖错误），就地修掉低级问题，形成"写→跑→看→改"的闭环，减少绕整个修复循环的成本。完整复现执行仍交由 execution agent。
- **execution agent（自适应的执行）**：从"一次性确定性管线"升级为能自主决策、自适应处理环境差异的 agent。它能在执行中途根据结果调整执行方式、针对环境差异做适配，而非只能机械重试或降级。
- **缺信息时可交互**：两个 agent 在执行中缺失任何关键信息（凭证、数据路径、参数、决策等）时，都能主动暂停流程向用户提问，拿到答复后继续（交互形态见 §5.3）。

#### 4.4.1 基于已有仓库的适配

当系统选择了一个可用的代码仓库时，编码工作聚焦于适配：

- **环境适配**：根据用户环境修改配置文件（如 CUDA 版本、Python 版本）
- **参数适配**：根据用户硬件调整训练参数（batch size、学习率等）
- **脚本补全**：补全缺失的评估脚本、数据下载脚本
- **Bug 修复**：处理仓库中已知的兼容性问题（参考 issue 信息）
- **版本适配**：处理依赖库版本不兼容的问题

#### 4.4.2 从零生成复现代码

当无可用仓库时，系统需要：

1. 基于论文方法章节和关键公式生成核心算法代码
2. 基于实验章节生成训练和评估脚本
3. 生成数据加载和预处理模块
4. 生成配置文件和环境依赖声明
5. 生成可运行的入口脚本和使用说明

代码组织应遵循清晰的项目结构：

```
reproduction_project/
├── README.md              # 使用说明
├── requirements.txt       # Python 依赖
├── configs/               # 配置文件
├── data/                  # 数据处理模块
├── models/                # 模型定义
├── train.py               # 训练入口
├── evaluate.py            # 评估入口
├── utils/                 # 工具函数
└── scripts/               # 辅助脚本（数据下载等）
```

#### 4.4.3 只编码不复现模式

> 决策日期：2026-05-06｜状态：**已决策**

当用户无法提供运行环境时：

- 系统完成上述编码工作，但不执行实验
- 交付物为完整的代码包和详细的执行说明文档
- 代码包应可在用户日后获得资源时直接使用

**code_only 模式交付标准采用"最低基准线 + agent 草拟 + 用户审核"机制：**

1. **最低基准线**（硬性要求，所有 code_only 交付必须满足）：
   - `README.md`：项目说明、复现步骤、依赖说明
   - `requirements.txt`：Python 依赖清单
   - 入口脚本（如 `main.py` / `train.py`）：可运行的主程序入口
   - 核心实现代码：论文核心方法的代码实现
   - 通过基础语法检查：代码无语法错误

2. **agent 草拟扩展**：planning 节点根据论文内容和资源情况，在最低基准线之上草拟更完整的交付标准建议（如配置文件、数据加载脚本、模型定义、评估脚本等）

3. **用户审核敲定**：用户在 planning 审核环节查看并确认/修改交付标准，最终交付标准以用户确认的版本为准。若用户不修改直接确认，则 agent 草拟的版本即为最终标准

### 4.5 测试与结果验证

#### 4.5.1 执行监控

- 系统启动实验后，应持续监控运行状态
- 关键监控指标：训练 loss 曲线、GPU 利用率、内存使用、预计剩余时间
- 运行过程中的日志应被完整保留

#### 4.5.2 结果验证

- **核心标准**：实验能跑出结果（程序正常结束，输出有效数值）
- **结果对比**：自动将复现结果与论文中报告的数值进行对比
- **差异分析**：当结果不一致时，系统需要分析可能的原因，包括但不限于：
  - 随机种子差异
  - 硬件差异导致的数值精度问题
  - 超参数配置差异
  - 数据集版本差异
  - 训练轮次不足
  - 代码实现差异（从零编码时）

#### 4.5.3 错误处理策略

> 决策日期：2026-05-06｜状态：**已决策**｜技术详见：技术架构文档 §12

**总体策略**：三层防御式错误处理架构（工具层重试 + 节点层降级 + 图层恢复），核心原则为"宁可带瑕前进，不可无故停下"。

**错误处理行为定义**：

| 错误类型 | 处理策略 | 用户感知 |
|---------|---------|---------|
| 网络超时/API 限流 | 工具层自动指数退避重试（3 次上限） | 无感知（底层自愈） |
| LLM 输出格式错误 | 节点层附带错误信息重试（3 次） | 无感知（底层自愈） |
| 论文章节读取失败 | 降级到全文提取 → 标记缺失 | 活动日志记录 + planning 审核页汇总展示 |
| 仓库搜索失败 | deepxiv github_url → Papers With Code → web search 逐级降级 → 零资源时切换 from_scratch | 黄色警告（WARNING） |
| 环境配置失败（pip install） | 逐包安装，失败包尝试降级版本 → 反馈 coding 寻找替代 | 执行监控页展示 |
| 编译/导入错误 | coding 节点自动修复循环（10 轮） | 执行监控页展示"自动修复状态" |
| 训练过程中崩溃 | coding 节点收到错误信息后修复代码并重试 | 执行监控页展示"第 N/10 轮修复" |
| OOM（内存/显存不足） | 归为硬件类不可自动修复错误：不进入自动修复循环，直接触发 interrupt#2 暂停等待用户决策，修复建议提示减小 batch size / 更大显存 / 切换 CPU | 执行监控页展示 |
| 执行超时 | 默认 30 分钟，SIGTERM → 5s → SIGKILL | 通知用户，可选择延长或终止 |
| 结果数值异常 | 不视为错误，在报告中分析原因 | 报告"差异分析"章节 |
| 数据下载失败 | 尝试备选下载源 → 反馈 coding 修改数据路径 | 执行监控页展示 |
| LLM API 认证失败 | 不重试，立即提示用户 | 模态对话框（ERROR） |
| git 仓库 / 模型 / 数据集凭证缺失 | 不重试不空转，agent 主动发起用户交互索要凭证（sp4），注入后继续 | 执行监控页交互输入框 |
| 论文不存在 | 不重试，提示用户重新输入 | 全页面错误（FATAL） |

**execution ↔ coding 自动修复循环（最多 10 轮）**：

execution 节点检测到代码执行失败后，将错误信息（stderr 关键行 + 上下文代码）写入状态，路由回 coding 节点让 LLM 修复代码，然后重新执行。每轮修复时 coding 节点收到：上次代码、具体错误、已尝试的修复方向（避免重复）、剩余次数。

**缺信息时的第三档处理（暂停交互，Sprint 4 新增）**：

除"自动重试"和"降级"外，当失败根因是"缺一条具体信息"（如凭证缺失、数据路径未知、需人工决策）时，agent 不再在修复循环里空转直到耗尽 10 轮，而是识别后主动暂停流程、向用户提问该信息，拿到答复后继续。凭证缺失（如 git clone 私有仓库读不到用户名）是该能力最迫切的首要用例——这类"问到就能继续"的失败不消耗修复循环次数。交互形态见 §5.3。

**10 轮修复失败后的用户决策**：

系统暂停流程，向用户呈现三个选项（不自动降级）：

| 选项 | 说明 | 适用场景 |
|------|------|---------|
| **A. 导出代码包 + 错误诊断报告** | 已生成代码、环境配置、详细错误诊断打包交付 | 用户有能力自行排查（小王场景） |
| **B. 回到计划审核，调整方案后重试** | 回退到 planning 审核点，展示失败原因，让用户修改计划后重新执行 | 错误源于计划层面（硬件不足、数据集不可用等） |
| **C. 终止任务** | 流程直接结束（checkpoint 与 workspace 产物保留，可追溯；不生成终止报告） | 用户决定暂时放弃自动复现（小张场景） |

**降级控制权分层**：

| 降级影响 | 控制方式 | 示例 |
|---------|---------|------|
| 低影响 | 自动降级 + 事后通知 | 章节读取切换到全文提取、deepxiv github_url 未命中切换到 Papers With Code / web search |
| 中影响 | 自动降级 + 实时警告 | 零资源切换 from_scratch、部分信息缺失 |
| 高影响 | 暂停等待用户决策 | execution 10 轮修复失败 |

所有降级信息在 planning 审核页面汇总展示为"信息完整度评估"卡片，确保用户确认计划前充分了解上游情况。

**错误信息展示规范**：

采用三层信息架构，面向 CS 学生用户"懂技术但不想读堆栈"的心智模型：

- **第一层（默认）**：一句话摘要 — "[做什么] 时 [发生了什么]，[系统做了什么 / 需要您做什么]"
- **第二层（可展开）**：节点名称、触发条件、已尝试方案及结果
- **第三层（链接）**：完整日志，保存在 workspace_dir/logs/ 下

措辞规范：不使用"系统异常""未知错误"；技术术语可用但不堆砌；所有错误信息必须给出"下一步"。

**用户通知分级**：

| 级别 | UI 表现 | 适用场景 |
|------|---------|---------|
| INFO | 活动日志流中的一行记录 | 自动恢复的瞬态错误 |
| WARNING | 黄色警告框，持续显示 | 降级运行的情况 |
| ERROR | 模态对话框，需用户选择"重试/跳过/终止" | 需用户干预的错误 |
| FATAL | 全页面错误 + 恢复操作按钮 | 流程完全中止 |

**重试预算**：单任务总预算最多 120 次 LLM 调用（config.py `MAX_TOTAL_LLM_CALLS`）；单节点 ReAct 轮次按节点差异化配置（paper_intake 5 / paper_analysis 12 / resource_scout 10 / planning 8 / coding 12 / execution 10），详见 technical-architecture.md §3.2.1 与 config.py `REACT_MAX_ROUNDS_*`。预算对用户不透明、不可调——这是工程保护措施，不是产品功能。预算接近耗尽时给出 WARNING 通知；最终报告中记录总修复尝试次数。

### 4.7 输出语言策略与本地化

> 决策日期：2026-05-17｜状态：**已决策**｜实施时机：Sprint 2（与 Streamlit UI 接入同步落地），Sprint 1 保持现状不动 prompt 主体。

#### 4.7.1 设计原则

paper_intake / paper_analysis 等节点的 system prompt 用中文撰写指令，但 LLM 输出的字段值（title / abstract / method_summary 等）延续工具返回的论文原文（英文）。这导致两个问题：

1. **UI 不友好**：CS 学生用户虽有英文阅读能力，但首页卡片、计划审核页等高频展示位的纯英文摘要会降低浏览效率
2. **若强制全中文**反而引发更严重的副作用：术语机翻歧义（如 "attention" 译"注意力"歧义）、下游 resource_scout 拿"中文数据集名"搜 GitHub 会搜不到、且会破坏 system prompt 主体的字节级稳定性进而退化 Prompt Cache 命中率（R-PC4）

**核心设计原则**：按字段的"主消费方"决定输出语言，而非"统一中英文"。**事实层字段（用于下游检索 / 匹配 / 配置）一律英文；UI 展示型字段提供中文；既给 LLM 又给 UI 的自由文本字段按下游 LLM prompt 语言对齐**。

#### 4.7.2 字段分类原则

将系统中所有结构化输出字段划分为五类：

| 分类 | 判定规则 | 输出语言规则 | 示例 |
|---|---|---|---|
| **A. 标识符 / 不可翻译值** | URL、ID、日期、数字、LaTeX、枚举值 | 原值不变 | arxiv_id, github_url, citation_count, key_formulas, framework |
| **B. 英文事实层** | 主消费方是下游节点用于检索、API 查询、配置匹配 | 强制英文，禁止翻译 | datasets, metrics, keywords, categories, sections_read |
| **C. 双语字段（中英并存）** | 主消费方既包含下游 LLM/检索，又包含 UI 展示 | 英文为主字段（保留原文用于检索/事实层），新增 `*_zh` 字段供 UI 使用 | title / title_zh、abstract / abstract_zh、tldr / tldr_zh |
| **D. 中文优先 + 英文备份** | 主消费方是中文 prompt 的下游 LLM 节点（planning / reporting），UI 也消费 | 主字段中文，新增 `*_en` 字段保留英文备份 | method_summary / method_summary_en、hardware_requirements / hardware_requirements_en |
| **E. 混合字段** | 自由文本含机器可读标签 | 自由文本用中文，机器可读标签（如 `[DEGRADED] missing=...`）保留英文语法不变 | analysis_notes |

#### 4.7.3 PaperMeta / PaperAnalysis 字段清单

**PaperMeta（步骤 1 输出）**：

| 字段 | 分类 | 输出语言 | 说明 |

| citation_count | A | 数字 | |

#### 4.7.4 Prompt 层面的实现指引

输出语言策略落到 prompt 层时，必须遵循以下硬约束（与 R-PC4 联动）：

1. **system prompt 主体冻结**：现有 `_INTAKE_SYSTEM_PROMPT` / `_ANALYSIS_SYSTEM_PROMPT_BODY` 主体内容已进入冻结期。**输出语言约束不可写入主体中段**，否则破坏 Prompt Cache 前缀稳定性
2. **语言约束放置位置（按优先级）**：
   - **首选**：写入 HumanMessage 通道（与现有 `_format_paper_context` 同通道）。例如 "请按以下语言策略输出：title 字段保留英文原文，title_zh 字段输出中文翻译，..."
   - **次选**：作为 system prompt **尾部独立段落**追加（不修改主体），通过明确分隔符（如 `--- 输出语言策略 ---`）与主体隔离
   - **禁止**：插入到 system prompt 中段或修改已有段落
3. **schema 同步更新**：`PAPER_META_SCHEMA` / `PAPER_ANALYSIS_SCHEMA` 必须扩展 `*_zh` / `*_en` 字段定义，否则 structured output 解析会丢字段
4. **字段缺失兜底**：当 LLM 漏写 `title_zh` 等翻译字段时，节点层 `_map_*_result` 必须有兜底逻辑（与 BUG-S1-02 / BUG-S1-03 同模式）：可选策略 a）`title_zh = title`（拉丁兜底）+ degraded 标记；b）二次轻量 LLM 调用专门做翻译。推荐策略 a），降级到 degraded_nodes 即可，避免额外 LLM 调用消耗预算

> **R-PC4 风险约束（强制）**：Sprint 1 dev-plan.md L885 已明确"system prompt 主体进入冻结期、动态变化走 HumanMessage 通道"。本决策的语言策略约束**禁止以任何形式修改 prompt 主体**；Sprint 2 实施前必须先通过 F 阶段实验 1 跑一次 Prompt Cache 命中率基线，作为前后回归对照。

#### 4.7.5 与下游节点的协作约束

## 5. 交互设计

### 5.1 技术选型

- **主界面**：Streamlit 轻量 Web UI
- **备选入口**：CLI（命令行接口，供高级用户使用）
- **架构原则**：核心逻辑与 UI 层完全分离，UI 仅为展示和交互层

### 5.2 页面流程

系统的 Streamlit UI 包含以下主要页面/视图：

#### 页面 1：论文输入页

- **核心元素**：
  - arXiv ID 输入框
  - 关键词搜索框
  - 搜索结果展示列表（含标题、摘要、引用数、GitHub 关联标识）
  - 论文确认按钮
  - "开始复现"按钮
- **交互流程**：
  - 用户输入 ID 或关键词 -> 系统在 UI 层调用 deepxiv_sdk 展示匹配结果 -> 用户浏览并确认目标论文 -> 用户点击"开始复现"按钮，将已确认的 arXiv ID 提交给后端 graph 启动后续流程
  - 所有论文搜索和确认操作均在此页面完成，后端 graph 的 paper_intake 节点仅接收已确认的 arXiv ID，不涉及用户交互

#### 页面 2：论文分析进度页

- **核心元素**：
  - 论文基本信息卡片（标题、摘要、作者）
  - 分析进度指示器（正在分析哪个章节）
  - 已发现的资源列表（实时更新）
- **交互流程**：
  - 自动展示分析进度 -> 用户等待完成（可查看中间结果）

#### 页面 3：复现计划审核页（核心交互页）

- **核心元素**：
  - 完整复现计划展示（分章节可折叠）
  - **候选仓库信息卡片**：展示 resource_scout 的候选仓库列表（仓库名、质量评分、选择理由），当前选用仓库高亮标识，用户可在此要求更换仓库
  - **信息完整度评估卡片**：当前置节点有降级时（degraded_nodes 非空），醒目展示各步骤信息完整度、降级原因和建议审核重点
  - 环境需求提示（与用户当前环境的匹配情况）
  - 用户环境配置表单（选择/输入 GPU 型号、内存等）
  - 计划调整区（用户可编辑或添加备注）
  - **code_only 交付标准卡片**（仅当用户选择"只编码不运行"模式时展示）：展示 agent 草拟的交付标准建议（在最低基准线之上的扩展项），用户可确认或修改，最终版本作为 coding 节点的交付依据
  - "确认执行"/"要求修改"/"只编码不运行" 三个操作按钮
- **交互流程**：
  - 用户阅读计划 -> 查看候选仓库选择（可要求更换，更换后计划将重新生成）-> 查看信息完整度评估（如有降级）-> 配置环境信息 -> 确认或要求修改 -> 进入执行或修改循环
  - 若用户选择"只编码不运行"：查看 agent 草拟的交付标准 -> 确认或修改交付标准 -> 进入编码阶段

#### 页面 4：执行监控页

- **核心元素**：
  - 当前执行步骤指示
  - 实时日志输出区（可滚动）
  - 关键指标仪表盘（如训练 loss、epoch 进度）
  - **自动修复状态区**：当系统进入 execution↔coding 修复循环时，展示"第 N/10 轮修复"、每轮错误摘要和修复策略
  - **错误信息展示组件**：一句话摘要 + 可展开详情 + 完整日志链接
  - 暂停/终止按钮
  - **10 轮修复失败后的用户选项**：A. 导出代码包+诊断报告 / B. 回到计划审核 / C. 终止任务（流程直接结束，checkpoint 与 workspace 产物保留）
  - **信息索取交互区（Sprint 4 新增）**：当 coding / execution agent 在执行中缺失关键信息时，页面弹出问题说明 + 单个输入框（凭证等敏感项用密码框输入并全程脱敏，并提供"记住此凭证供后续复现复用"勾选），用户填写后提交，流程从暂停处继续
- **交互流程**：
  - 系统自动执行 -> 用户可实时查看进度 -> 出错时系统自动修复（最多 10 轮）-> 修复成功则继续 -> 修复失败则暂停并展示三选项
  - 执行过程中若 agent 缺失关键信息（凭证 / 数据路径 / 决策等）-> 流程暂停并在本页弹出信息索取交互区（逐条提问）-> 用户填写提交 -> agent 拿到信息后从暂停处继续；在没有交互前端可应答时，按信息缺失降级处理而非无限等待

#### 页面 5：结果与报告页

- **核心元素**：
  - 结果摘要卡片（核心指标一览）
  - 与论文对比表格（差异高亮显示）
  - 完整报告查看区
  - 报告下载按钮
  - 代码包下载按钮
- **交互流程**：
  - 自动展示结果摘要 -> 用户查看详情 -> 下载报告或代码

### 5.3 执行过程中的信息索取交互（Sprint 4 新增）

> 决策日期：2026-07-01｜状态：**已决策（Sprint 4，路线丙）**｜详见 `docs/sprint4/prd.md`

在此之前，系统的用户交互只覆盖三处：论文输入页的论文确认、复现计划审核页的计划审核、执行监控页 10 轮修复失败后的 A/B/C 决策。Sprint 4 新增第四种交互形态——**执行过程中 agent 主动向用户索取信息**：

- **触发场景**：coding / execution agent 在执行中缺失任何推进所需的关键信息（凭证 / 数据路径 / 参数 / 需人工决策的选择），最典型的是凭证缺失。
- **交互位置**：在执行监控页（页面 4）就地承载，不新增独立页面；表现为问题说明 + 单个输入框。
- **交互规则（已决策）**：
  - **逐条提问**：一次只问一个信息项，避免一次性抛出大表单。
  - **不设硬超时**：流程可一直暂停等待用户，任务进度天然保留；不会因等待过久而自动失败。
  - **敏感信息保护**：凭证类信息用密码框输入、全程脱敏展示，可选"记住供后续复现复用"（记住后同类信息在后续复现中不再重复询问）。
  - **无交互前端时降级**：在 CLI / 自动化等没有交互前端可应答的场景，按信息缺失降级处理，而非无限等待。
- **与既有交互的关系**：与论文确认、计划审核、修复失败决策三种交互并列、互不干扰，同属"人在回路"的产品能力。

### 5.4 CLI 入口设计

CLI 入口提供与 Web UI 等价的功能，适合高级用户和自动化场景：

```bash
# 启动复现任务
auto-repro run --paper 2409.05591

# 指定配置
auto-repro run --paper 2409.05591 --env local --gpu "RTX 3090"

# 查看计划（不执行）
auto-repro plan --paper 2409.05591

# 只编码不执行
auto-repro code-only --paper 2409.05591

# 查看任务状态
auto-repro status --task <task_id>
```

## 6. 平台与部署策略

### 6.1 三种运行模式

| 模式 | 描述 | 适用场景 |
|------|------|---------|
| **本机模式** | 在用户的本地机器或实验室服务器上执行复现 | 用户拥有足够算力的本地设备 |
| **云服务器模式** | 在用户自行提供的云服务器上执行复现 | 用户有云服务器账号但不想在本地跑 |
| **只编码模式** | 系统只生成代码，不执行实验 | 用户暂无可用的运行环境 |

### 6.2 本机模式

- 系统运行在用户本地
- 自动检测本地硬件配置（GPU 型号、显存、CPU、内存）
- 根据硬件约束自动调整复现参数
- 所有数据和代码保留在本地

### 6.3 云服务器模式

- 用户提供云服务器的连接信息（SSH 地址、密钥等）
- 系统通过远程连接在云服务器上执行复现
- 需要用户自行完成云服务器的基础配置（操作系统、驱动）
- **待确认：远程执行的具体技术方案（SSH 隧道 / 远程 agent 部署 / 其他）**

### 6.4 只编码模式

- 系统完成论文分析、资源搜集、复现规划、代码生成全流程
- 输出完整的代码包，包含：
  - 所有源代码文件
  - 环境依赖文件（requirements.txt / environment.yml）
  - 详细的执行说明文档（README）
  - 配置文件和脚本
- 交付标准采用"最低基准线 + agent 草拟 + 用户审核"机制（详见 §4.4.3）

### 6.5 系统自身的部署

系统本身（LangGraph 编排核心 + Streamlit UI）运行在用户的本地机器上。用户通过浏览器访问 `localhost` 上的 Streamlit 界面。

### 7.3 MVP 验收标准

1. 用户输入一个有 GitHub 关联仓库的 CS 论文 arXiv ID，系统能自动完成从分析到报告的全流程
2. 复现计划页面可正常展示和修改
3. 在本机环境下能成功执行实验并跑出结果
4. 报告中包含复现结果与论文数据的对比表
5. 当结果不一致时，报告中有原因分析
6. Streamlit UI 五个核心页面均可正常使用
7. 端到端完成一次复现的时间（不含训练时间）在合理范围内
