## 0. Sprint 主题与立项依据

### 0.1 立项依据：approve 之后的"最后一公里"整体失守

### 0.2 Sprint 6 主题

**交互贯通与任务可恢复**：sp5 解决"系统对用户诚实"，sp6 解决"系统对用户到场"——用户的每次应答有回音（**不死锁**）、用户的每个决策被记住（**不失忆**）、用户的每个任务找得回（**不失联**）。

面试叙事价值（按 Maria 一等权重标注，不为凑叙事扩需求）：**S6-01**（Streamlit 无推送模型下的异步 HITL 过渡态状态机设计）、**S6-03/04/05**（agent 上下文工程 + 确定性守门，治 LLM 规避形态演化"编造→躺平→两面"，为 Sprint 7 评测失败归因供分类地基）、**S6-07**（LangGraph checkpoint 会话恢复能力外化）为本 Sprint 三个最高叙事价值点。

### 2.2 范围边界（防发散）

- **主图 7 节点骨架不变**（沿用 sp3/sp4/sp5 同范式）：全部需求落在 UI 层 / controller 层 / 编排收尾层 / prompt / 工具回调层 / schema，不新增节点、不改节点集合。
- **不新增 interrupt 种类**：现有 interrupt#1（计划审核）/#2（dev_loop 决策）/#3（user_input，含凭证 gate）三种通道封口；S6-05 的数据可得性警示复用审核交互，S6-03 的短路是"减少 interrupt"而非新增。
- **判定逻辑归确定性代码**（沿用 sp5 三条产品红线之一）：过渡态换代判定、降级短路判定、no_metrics 判定、计划交叉检查、任务状态推导，全部为确定性逻辑，不交 agent 自由裁量。

- **沿用轮询**：不上 WebSocket/SSE/事件推送重构，过渡态复用既有 st_autorefresh 节奏。
- **prompt 改动全部遵守 R-PC4 前缀冻结**（HumanMessage 通道或 system prompt 尾部独立段落）；sp5 前缀冻结令延续，Prompt Cache 基线复采随收口真跑窗口。

## 3. 需求详述

### S6-01 HITL 面板提交后过渡态统一契约（卡点 A，P0）

- **问题**（走查 §二-A 已核实机制）：`ui/pages/execution_monitor.py` 两类面板（user_input `_render_user_input_panel` :713/:733、dev_loop 决策 `_submit` :535）提交/降级后走 `resume_with + st.rerun()`；rerun 瞬间 worker 尚未消费 resume，`is_interrupted` 仍真 → 再次命中 case⑤ 重画原面板，而 case⑤ 是"停轮询"分支（`st_autorefresh` 仅 case⑦ 注册）→ 无任何后续 rerun，UI 永久冻结；再点一次 = 对新 interrupt 的误提交（双重 resume 风险）。竞态窗口 = worker 重放节点到 interrupt 点全程（秒~分钟级），实际必中。dev_loop `_submit` 注释"提交后 st.rerun() 让本页轮询自愈"与实现矛盾。对照：plan_review 的 approve 有 awaiting 轮询兜底（走查 approve→gate 5s 转场正常），是本项的既有参照范式。
- **需求**（行为契约，不预设实现）：
  1. 两类面板提交（含降级）后，用户必须**立即获得"已收到、处理中"的可感知反馈**，且页面自动持续刷新，直至流程状态变化——**不允许存在"停轮询且等待后台变化"的静默驻留状态**。
  2. 过渡期间原面板不可重复提交；**同一次用户提交至多产生一次 resume**。
  3. 过渡态退出条件二选一：interrupt 消失（转正常监控/终态/跳转）或 interrupt **换代**（出现新一条提问 → 渲染新面板）；"换代"须有确定性判定依据（交架构师，Q-S6-1）。
  4. 将「render() 分发中，凡'等待后台变化'的状态必须注册轮询；'停轮询'分支只允许承载'仅用户动作可改变'的状态」确立为本页设计通则，落档到代码注释/文档（顺带修正 dev_loop :535 的矛盾注释）。
- **文件级修改点**（供开发对位，非实现规格）：`ui/pages/execution_monitor.py`（case⑤ 两面板 + render() 分发注释）；参照 `ui/pages/plan_review.py` awaiting 范式。
- **面试价值**：异步 UI↔worker 的 HITL 过渡态一致性设计——建议完成后与 S6-06/07 合并产出一份 docs/reports/ HTML report（"HITL 与会话恢复"）。
- **验收口径**：AC-S6-01 ~ 03。

### S6-02 在途阶段指示即时化 + 在途自动切监控页（P1）

- **问题**（TODO 07-10）：current_step 由节点完成时写入 → coding 运行期间 UI 仍显示"制定计划"，标签滞后一整个节点；进度页 case④bis 依赖该标签，故在途也不自动切监控页。与卡点 A 同族（sp5 只修终态/interrupt 路由，未修在途显示）。

  2. 进度页在 coding/execution 在途时自动切换到执行监控页（case④bis 真实生效）。
- **验收口径**：AC-S6-04 ~ 05。

### S6-03 用户降级决策全链路贯穿（卡点 C，P0）

- **问题**（走查 §二-C 已核实）：用户在 gate 显式选择"无此凭证，降级为模拟实验"后，coding 仍产真实调用 OpenAI 的代码 → 修复循环空烧 2 轮真配额 → execution agent 经 interaction 工具**再次 interrupt 索要同一凭证**。checkpoint 取证 `degraded_nodes=[]`——降级未记账。降级语义没有进入 coding prompt / execution 上下文，agent 对"用户已拒绝该凭证"零记忆，理论上可无限循环要钥匙+烧配额。**测试盲区警示**：sp5 AC-S5-03 曾以 mock e2e 断言"降级标记全链路可见"并通过，真跑记账为空——本项验收必须以走查现场同构场景为回归靶，不得只靠 mock 自证。
- **需求**：
  1. 用户在 gate 的降级决策**确定性记账入 state**（最小形态：被拒凭证标识清单；字段名与落点交架构师 Q-S6-2）。
  2. 该决策**注入 coding 与 execution 的 agent 上下文**：产码必须走不依赖该凭证的路径（模拟/mock），修复循环每轮上下文同样可见该决策。
  3. **interaction 工具对已拒绝凭证确定性短路**：对同一已拒绝凭证的再次索要不产生 interrupt，直接返回降级指令。产品红线：**用户显式拒绝一次 = 全任务生效**（撤销通道不做，见非目标 4）。
  4. 记账贯穿至报告：复用 sp5 的降级强制声明链路（S5-01/S5-04 既有通道），不新增第二套标注体系。
- **边界**：贯穿的是"用户显式决策"这一结构化信息，不引入 agent 间自由对话通道；prompt 注入遵守 R-PC4。
- **面试价值**：agent 上下文工程——"决策记忆"防退化循环，与 S6-04 合并即"关键信息贯穿"完整叙事。
- **验收口径**：AC-S6-06 ~ 08。

### S6-04 no_metrics 零指标专属处置 + 修复循环定向引导（P1）

- **问题**（TODO 07-10 实锤，thread task-19e21e015017）：对账 10/10 步全 exit 0 但 `<METRICS>` 与 outputs/summary.json 均无产出 → success=False 走 dev_loop 决策面板，错误摘要却是 `[error_category=none] 执行成功`——"无错误"与失败决策同屏自相矛盾；修复循环健康跑了 2 轮（修 import、修路径）修完仍栽在同一判定上再弹窗——coding 收不到"缺的是指标输出"这个信息，每轮空烧真实配额。
- **需求**：

### S6-05 计划自洽确定性交叉检查 + 数据可得性审核警示（P1）

- **问题**（TODO 07-10 三层根因取证）："两面计划"——叙述面诚实丰满（承认无可复用仓库、data_preparation 列 6 条、如实报耗时），但与 execution_steps 完全脱节（10 步全搭骨架、无一步跑实验出指标、真实交付标准藏在 deliverables 末条 py_compile），审核者被叙述面引导过审。根因链：论文太新无资源 → 数据本地缺失且规划模型自知 → sp5 禁编造后 LLM 规避形态从"编造"转为"躺平/两面"。体系空档 = **数据可得性无前置暴露机制**（凭证有 gate、数据没有）。
- **需求**：
  1. **计划确定性自洽交叉检查**（纯规则，非 LLM），两条起步：①data_preparation 非空 ∧ 执行步骤无对应数据步骤 → 警示；②expected_results 含指标性预期 ∧ 执行步骤无产出指标的步骤 → 警示。警示展示于计划审核页既有"信息完整度评估卡片"位（复用，不新增卡片体系）。**警示不阻断审批**——用户知情后仍可批准（人在回路本义）。
  2. **数据可得性暴露**：资源侦察无可用数据集 ∧ 计划声明需数据 → 审核页明确警示"所需数据集不可得，请决策"（复用审核交互与上条同位展示，**不新增 gate/interrupt 种类**——凭证 gate 独立存在是因为它在 approve 之后才被消费且需要 secrets 输入通道，数据缺口在审核时点即可决策；见假设 A-S6-2）。

- **边界**：规则只做上述条目起步，不做计划质量评分体系；**误报防线与命中防线同权重验收**（自洽计划零警示）。
- **面试价值**：确定性守门治 LLM 规避形态演化（编造→躺平→两面），为 Sprint 7 评测失败归因供前置分类地基。

## 4. 范围悬案裁决：评测体系与 token/成本追踪 → Sprint 7 合成立项

**裁决**：两个既有候选（TODO :363-364）**双双不并入 sp6**，维持"两者合成一个 Sprint"的已确认结构（Q-S5-1，Maria 2026-07-08 确认），**编号顺延为 Sprint 7**；sp6 定位为其**第二道硬前置**。理由：

1. **因果顺序**：跑批评测的前提是单篇正常流程走得通。走查证明 approve 后即死：卡点 A（死锁）使人在回路跑批无法完成，卡点 C（降级不贯穿、修复循环空烧）使成本数字失真，卡点 D（失联）使长任务无法看护。先修通道再跑批，与 sp5"先把判定口径做诚实、再统计成功率"（sp5 PRD §4.2）完全同构——否则 Sprint 7 产出的成功率表与成本表会被缺陷污染，数字不可信。
2. **体量**：sp6 已承载三范式 + 7 项机械修复（8 项需求、4 项 P0、23 条 AC），量级与 sp5 相当；再叠"3-5 篇 benchmark 跑批 + token 计量 + tracing 接入"会挤压 P0 闭环质量。
3. **面试价值不受损反增**：Sprint 7 的成功率表 + 成本表建立在"诚实判定（sp5）+ 可走通流程（sp6）"之上，叙事更完整；且 sp6 为 Sprint 7 预置了失败归因分类地基——no_metrics 专属类别、计划自洽检查警示、降级记账，都是逐篇失败归因的现成维度（另有 t53 留档"degrade 产码方差入评测统计"候选）。**但 sp6 不为 Sprint 7 预留任何投机性扩展点**（极简红线，沿 sp5 §4.1 先例）。
4. **配额现实**：评测跑批耗真实配额，须独立授权窗口与省配额设计，与 sp6 修复流混排互相拖累。

与 Maria 已确认裁决的关系：Q-S5-1 的实质是"评测体系与 token/成本追踪合成一个 Sprint、且 sp5 是其前置"——本裁决完整保留该实质，仅因走查插入阻断级缺陷 Sprint 而编号顺延（假设 A-S6-1，推翻成本低：若 Maria 要求并入，评测部分可作为 sp6 后置批次追加，不影响本 PRD 既有需求）。
