# 全局 PRD（docs/product-design-specification.md）Sprint 6 立项审查结论

> 产出：产品经理代理，2026-07-13（随 sp6 PRD v1.0 一并产出；Maria 不在线，按硬约束一次性产出）
> 状态：**待应用**。建议 Maria 回线确认 sp6 PRD 后，由主控一次性批量应用（v0.3 → v0.4），沿 sp5 R1~R13 批量应用先例；R13 更新日志行必须与 R1~R12 同批落。

总体结论：2026-07-12 真实浏览器走查暴露的事实与全局 PRD 两处承诺直接相悖——"可交互推进"承诺只兑现了"提问"半程（应答提交后 UI 永久死锁、agent 重复索要已拒凭证），"过程透明"在面板提交后的过渡窗口失效；"任务可恢复"在产品能力清单中整体缺位（刷新即失联却无任何页面/交互承载重连）；另有 Papers With Code 通道已死造成文档与事实脱节。共 13 项修改点（R1~R13：10 项【产品决策】+ 2 项【事实对齐】+ 1 项元数据）。

## R1｜§1.3 核心价值主张·"可交互推进"行｜【产品决策】

- 原文：`| 可交互推进 | 编码与执行阶段的 agent 在缺失关键信息（凭证、参数、决策等）时，能主动暂停流程向用户提问，拿到答复后继续，而非空转或直接失败 |`
- 改为：`| 可交互推进 | 编码与执行阶段的 agent 在缺失关键信息（凭证、参数、决策等）时，能主动暂停流程向用户提问，拿到答复后继续，而非空转或直接失败；用户应答提交后界面即时呈现处理进展并自动恢复推进（不允许静默冻结），用户已明确拒绝提供的信息（如选择降级）在整个任务内全链路生效，系统不得重复索要 |`
- 理由：走查实锤该承诺只兑现前半程——降级提交后 UI 永久冻结 15+ 分钟（卡点 A），agent 对已拒绝凭证再次索要（卡点 C）。"提问→应答"闭环必须把"应答后的贯通"写入承诺，否则价值主张与 sp6 修复口径脱节。

## R2｜§1.3 核心价值主张·新增"任务可恢复"行（追加在"可交互推进"行之后）｜【产品决策】

- 新增：`| 任务可恢复 | 任务进度持久化于 checkpoint；浏览器刷新、重开或链接直达均可重连既有任务，运行中/等待输入的任务不因界面会话中断而失联（任务列表一键挂回） |`
- 理由：卡点 D 证明"长流程 + 人在回路"产品若无重连能力，一次 F5 即湮灭任务（checkpoint 里挂着 interrupt 等人，UI 层永远找不回）。现有五条价值主张无一覆盖恢复能力——这是产品能力缺位而非实现瑕疵，须升格为价值主张。

## R3｜§5.2 页面 4：执行监控页·信息索取交互区 bullet 与交互流程 bullet（两处）｜【产品决策】

- 原文（其一）：`  - **信息索取交互区（Sprint 4 新增）**：当 coding / execution agent 在执行中缺失关键信息时，页面弹出问题说明 + 单个输入框（凭证等敏感项用密码框输入并全程脱敏，并提供"记住此凭证供后续复现复用"勾选），用户填写后提交，流程从暂停处继续`
- 改为（句末追加）：`…用户填写后提交，流程从暂停处继续；提交后交互区立即转入"处理中"过渡态并自动刷新，直至流程继续或出现下一条提问，同一提问不会被重复提交（Sprint 6）`
- 原文（其二）：`  - 执行过程中若 agent 缺失关键信息（凭证 / 数据路径 / 决策等）-> 流程暂停并在本页弹出信息索取交互区（逐条提问）-> 用户填写提交 -> agent 拿到信息后从暂停处继续；在没有交互前端可应答时，按信息缺失降级处理而非无限等待`
- 改为：`  - 执行过程中若 agent 缺失关键信息（凭证 / 数据路径 / 决策等）-> 流程暂停并在本页弹出信息索取交互区（逐条提问）-> 用户填写提交 -> 界面转入"处理中"过渡态并自动刷新 -> agent 拿到信息后从暂停处继续；在没有交互前端可应答时，按信息缺失降级处理而非无限等待（Sprint 6：提交后过渡态可感知，杜绝静默冻结）`
- 理由：卡点 A 的页面级承载。同页的 10 轮修复失败决策面板适用同一契约（sp6 S6-01 两类面板统一），页面定义以交互区两处表述为锚即可，不逐条重复。

## R4｜§5.2 页面流程·新增"页面 0：任务列表与重连"（插入在"#### 页面 1：论文输入页"之前）｜【产品决策】

#### 页面 0：任务列表与重连（Sprint 6 新增）

- **核心元素**：

## R5｜§5.3 执行过程中的信息索取交互·触发场景 bullet｜【产品决策】

- 原文（现行文末句）：`…用户可显式选择"无凭证降级为模拟"，该降级将全链路标注并在报告中强制声明（agent 不得自行决定降级）。`
- 改为（句末追加）：`…（agent 不得自行决定降级）。Sprint 6 起，用户的拒绝/降级决策在任务内全链路贯穿：coding / execution agent 上下文获知降级指令并产出不依赖该信息的路径，系统对同一已拒绝信息的再次索要予以确定性短路（不再重复提问）。`
- 理由：卡点 C 实锤"全链路标注"承诺仅落在报告标注面，未进 agent 上下文——走查中 coding 仍产真实 API 调用码、execution 再次索要同一凭证、`degraded_nodes=[]` 未记账。贯穿语义须写全为产品承诺。

## R6｜§4.5.3 错误处理策略·错误处理行为定义表新增一行（"结果数值异常"行之后）｜【产品决策】
