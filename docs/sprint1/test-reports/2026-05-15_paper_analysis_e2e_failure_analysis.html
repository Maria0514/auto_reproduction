5. 排查步骤与根因定位

5.1 排除项

  1. 工具调用失败？ → INFO 日志显示 6 次 read_section 全部 success。排除
  2. ReAct 子图 finalize 解析失败？ → 其它字段全部完整填充，说明 <result> JSON 被正确解析。排除
  3. 类型补齐 _coerce_str_list 把内容强转空？ → 该函数只过滤 None / 空字符串，对非空字符串列表保持原样。LLM 输出本身就是空数组。排除
  4. 同 BUG-S1-02 的 tool 序列化回归？ → 工具结果实际到达 LLM 上下文（method_summary / datasets / metrics 都基于章节内容填好了，说明 LLM 看到了完整的章节文本）。本次问题不在序列化。排除

5.3 根因判定

分两层：

  • 诱因（LLM 服从度）：C2 的 system prompt 在"字段填充优先级"段明文规定 sections_read：实际成功读取的章节名列表（与工具调用历史一致），但 LLM 偶发会输出空数组——尤其在 baseline_results 字段填充得非常大（HippoRAG 的 baseline tables 占用了输出 token 的大头）、模型注意力被消耗的场景下。
  • 放大器（兜底缺失）：_map_analysis_result 当前只用 2 参签名（result, state），没用上 _make_react_wrapper 已经支持的"第 3 参 react_messages 透传"能力。架构 §2.8.2 "head 优先回填"契约要求节点在 LLM 漏字段时从 ReAct 子图工具历史回填——paper_intake 已通过 BUG-S1-02 修复的 _backfill_paper_meta_from_tools 实现了 categories 兜底；paper_analysis 没有对应的 sections_read 兜底。
判定：BUG-S1-03 是生产代码偶发 bug，不是测试用例问题。测试断言基于 _missing_core_fields 把 sections_read 列入核心字段——这是 C2 自己定义的契约，且 PRD / dev-plan 都要求该字段非空。

6. Bug 报告

Bug IDBUG-S1-03
归属节点core/nodes/paper_analysis.py
形态LLM 偶发漏写 sections_read 字段值（输出空数组），节点缺少工具历史回填兜底
复现率≈ 25%（4 次全量回归中 1 次复现）
复现命令pytest -v（重复跑约 4 次能复现一次）
或 pytest tests/test_paper_analysis_e2e.py::test_e2e_basic_path_full_pipeline -m e2e -v
影响范围paper_analysis 节点偶发误标记 degraded；污染 degraded_nodes / node_errors；阻塞 paper_analysis 单节点 e2e 稳定性
是否阻塞 C2 验收不阻塞主路径功能正确性（其它字段填充正常），但阻塞 paper_analysis 单节点 e2e 验收的稳定性

建议修复方向（按 ROI 排序）

  1. 首选：在 _map_analysis_result 加 ReAct messages 工具历史回填——
    • 把签名扩到 3 参（result, state, react_messages）；
    • 扫描 react_messages 中所有 name=read_section 的 ToolMessage（兼容失败 ToolMessage，需判定 content 不是 Error in ... / tool ... raised ... 前缀）；
    • 从对应的 tool_call args 抽 section_name（注意 tool_calls 在前一条 AIMessage，需配对扫描）；
    • 若 LLM 输出 sections_read 为空但工具历史非空，则用工具历史回填；
    • 同步在 analysis_notes 记录 [BACKFILL] sections_read from tool history。
    _make_react_wrapper 第 877 行已支持 3 参签名，无需改基础设施。
  2. 次选：强化 prompt——在"输出要求"段补一句"若调用过 read_section 且返回非错误，最终 JSON 的 sections_read 必须包含对应章节名，缺少视为违规输出"。受限于 LLM 服从度，不如 #1 稳定。
  3. 可选：把 sections_read 移出核心字段——降级判定不再依赖该字段。但这弱化了"读了哪些章节"的可观测性，不推荐。

修复后回归命令

for i in {1..5}; do pytest -v || break; done

至少连续 5 次 26/26 通过才可关 Bug——25% 复现率需要足够样本量才能确认治理。
