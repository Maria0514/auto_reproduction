一句话结论：core/tools/deepxiv_tools.py 的工具工厂把 dict 用 str(result)（Python repr，单引号）写入 ToolMessage.content， 但 core/react_base.py::extract_last_tool_result 用 json.loads() 解析—— 单引号不是合法 JSON，解析永远返回 None，导致 _backfill_paper_meta_from_tools 兜底从未对 deepxiv 工具生效； 当 LLM 漏写 categories 时（schema-enforce regen 也救不回来），最终输出 categories=[]。 这是生产代码 bug，不是测试用例问题。

3.3 流水线推演（按代码逐步走一遍）

  1. 第 1 轮 reasoning：LLM 决定先调 get_paper_brief，然后 get_paper_head。 工具均成功，head 返回字段含 categories=['cs.CL', 'cs.AI']（直接打 deepxiv 验证过）。
  2. 第 2 轮 reasoning：LLM 输出 <result>{...}</result>， 但 categories 字段值是空数组 []（LLM 服从度抽风，输入是否需要清洗都可能出现）。
  3. finalize_node 检测到必填字段缺失（core/react_base.py:580， _missing_required_fields 把空 list 视为缺失）→ 调用 _invoke_with_schema 重生成（core/react_base.py:591-596）。
    日志：finalize: schema-enforce regen (parsed=ok, missing=['categories'])。
  4. schema-enforce regen 也救不回 categories：因为重生成的 messages 上下文与第一次一样（head 工具结果以 Python repr 形式塞在 ToolMessage 里），且 schema 只要求字段类型为 array，不强制非空。 LLM 仍可能输出 categories: []。重生成结果通过 merged.update(forced) 覆盖了原解析（core/react_base.py:602），categories 依然空。
  5. 进入 _map_intake_result（core/nodes/paper_intake.py:230）：
    • 构造 paper_meta（_build_paper_meta）—— categories=[]。
    • 调 _backfill_paper_meta_from_tools（paper_intake.py:272）尝试从 head 工具历史回填。

3.4 根因定位：extract_last_tool_result 永远无法解析 deepxiv 工具的 ToolMessage

关键证据 —— 在 ReAct 真实序列化路径上重放 head 工具：

print('TOOL invoke type:', type(raw_str))

def _truncate(text: str) -> str:
    if len(text) <= TOOL_RESULT_MAX_LENGTH:
        return text
    return text[:TOOL_RESULT_MAX_LENGTH] + f"\n... [truncated at {TOOL_RESULT_MAX_LENGTH} chars]"

def get_paper_head_tool(token=None) -> BaseTool:

    @tool

        """..."""
        try:
            result = client.get_paper_head(arxiv_id)   # result 是 dict

4. 修复建议

本次任务只做诊断，不修代码。下列修复方向按 ROI 排序，建议交给 @全栈开发代理。

4.1 必须修：工具序列化用 JSON（根因 #1）

修改 core/tools/deepxiv_tools.py 的 7 个工具工厂，把 dict / list 用 json.dumps(...) 而非 str(...) 写入返回值。 注意保持 Prompt Cache 字节级幂等（同输入同输出文本），所以要固定 ensure_ascii、不要带 indent、不要带时间戳。

diff 思路（以 get_paper_head_tool 为例）：

# core/tools/deepxiv_tools.py
import json

def _serialize(value) -> str:
    """稳定序列化：dict / list 走 JSON；str / 其它走 str()。"""
    if isinstance(value, (dict, list)):
        return json.dumps(value, ensure_ascii=False, sort_keys=True, default=str)
    if isinstance(value, str):
        return value
    return str(value)

def get_paper_head_tool(token=None) -> BaseTool:
    client = DeepxivTools(token=token)

    @tool
    def get_paper_head(arxiv_id: str) -> str:
        """..."""
        try:
            result = client.get_paper_head(arxiv_id)
-           return _truncate(str(result))
+           return _truncate(_serialize(result))
        except Exception as exc:
            return f"Error in get_paper_head: {exc}"

    return get_paper_head

其它 6 个工厂同样替换（brief / structure / search / web_search 三个 dict-list 工具必改； read_section / get_full_paper 返回 str，可保持原样但建议统一走 _serialize 以防上游改类型）。

修复后，extract_last_tool_result 第一条 json.loads 路径直接命中， _backfill_paper_meta_from_tools 即可拿到 head dict 并兜底 categories / authors / abstract。

4.3 可选：extract_last_tool_result 加 ast.literal_eval 兜底

作为深度防御，core/react_base.py 的 extract_last_tool_result 在 JSON 全部失败后， 可以尝试 ast.literal_eval(text) 兼容 Python repr。但不建议作为首选修复，因为：

  • 会掩盖工具序列化不规范这一上游问题。
  • ast.literal_eval 对 set、tuple、自定义类的 repr 同样会失败，覆盖率有限。
  • 该 helper 是"基础设施级"，下游所有节点共用；与其每个调用点容忍乱序列化，不如约束所有工具走 JSON。

4.4 可选：finalize 的 schema-enforce regen 应明示"非空 list"

core/nodes/paper_intake.py:23-47 的 PAPER_META_SCHEMA 把 categories 标为 array of string， 但 JSON Schema 默认允许空数组。可在 schema 里加 "minItems": 1：

"categories": {"type": "array", "items": {"type": "string"}, "minItems": 1}

但注意：

  • OpenAI method=json_schema 对 minItems 支持有限，function_calling 路径可能直接忽略约束。
  • 论文若真的没有 categories（理论可能），强制 minItems 反而会让 LLM 编造分类。
  • 这条不能替代 4.1，应作为补强。

4.5 测试侧后续动作

  • 不修改 tests/test_paper_intake_e2e.py 的断言以"绕过"——契约就是 categories 非空。
  • 建议补单元测试 tests/test_paper_intake.py::test_backfill_categories_from_head_tool_message， 使用纯 mock 的 ToolMessage(content=<JSON 序列化的 head dict>) 验证 backfill 路径， 避免下次回归还要靠 e2e 才能发现。
  • 修复后回归命令：pytest tests/test_paper_intake_e2e.py -m e2e -v --durations=0， 连续 3 次全绿才可关 BUG-S1-02（避免被 LLM 服从度的偶发性误判为已修复）。
