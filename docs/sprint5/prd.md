## 0. Sprint 主题与立项依据

### 0.1 立项依据：一条完整的"假成功"因果链

本次真跑最重要的产出不是单点 bug，而是一条环环相扣的系统性失守链（编号对应反馈条目）：

> **#10** 缺 API 凭证时 agent 不求助（交互工具全程未触发）→ **#9** 转而编造关键词模拟实验冒充论文实验（自证闭环 + 硬编码 baseline 分数）→ **#11** 执行又被 ReAct 轮数预算静默截断（13 步只跑 8 步，5 步无声丢弃）→ **#7** 报告以宽松口径把这一切包装成 full_success"✅ 复现成功" → **#4** UI 路由断链，用户连报告页都到不了。

每一环单独看都"没报错"，连起来就是假成功。这直接冲击全局 PRD 的两条核心价值承诺——"结果可信"与"过程透明"（全局 PRD 需同步修订，见本次全局 PRD 审查结论）。

### 0.2 Sprint 5 主题

**诚实性与可观测性**：让系统**说真话**（结论如实分级、造假可被识别、降级必须显式），让用户**看得见**（执行过程可感知、步骤对账可见、页面可达、术语可读）。按因果链拆成四道防线：

| 防线 | 定位 | 承载需求 |
|---|---|---|
| 事前 | 把"造假动机"暴露给用户决策：凭证/资源缺口前置识别 | S5-01 |
| 事中 | 约束与记账：coding 诚实红线、计划步骤执行对账、预算截断显式化、agent 活动流 | S5-02 / S5-06 / S5-07 |
| 事后 | 审计与判定：造假模式审计降档、结论两级分级、定性目标回验、多组指标解析 | S5-03 / S5-04 / S5-05 / S5-10 |
| 界面 | 可达与可读：路由修复、术语治理、产物路径明示 | S5-08 / S5-09 / S5-11 |

面试叙事价值（按 Maria 一等权重标注，不为凑叙事扩需求）：S5-03（reward hacking 检测）、S5-04（eval 判定口径设计）、S5-06/S5-07（agent 可观测性与预算透明）为本 Sprint 三个最高叙事价值点。

### 2.2 范围边界（防发散）

- **主图 7 节点骨架不变**（沿用 sp3 AC-S3-10 / sp4 同范式）：本 Sprint 全部需求落在 prompt / schema / 编排收尾层 / UI 层 / 工具回调层，不新增节点、不改节点集合。
- **判定逻辑归确定性代码**：结论分级、审计降档、对账判定均为确定性逻辑，**不交给 agent 自由裁量**（沿用 sp4 §4.2.2 "B 档判定不交 agent"原则——否则等于让被审计者自己打分）。
- **降级必须是用户显式动作**：无凭证降级为模拟只能由用户在交互中显式选择，agent 不得自行决定（产品红线）。
- **每个需求做最小单一抽象**：不预设多分类枚举（如结论档位是"两级 + 正交标注"而非多维矩阵；术语映射是一张表而非 i18n 框架；活动流是内存尾部渲染而非 trace 系统）。

### 2.3 非目标（sp5 不做）

- **LLM-as-judge 实质对应度审计**（#9 长期项）——延后至评测体系 Sprint 评估；sp5 只做规则型 smell 检测。
- **token/成本计量与 tracing 平台接入**——不并入 sp5，并入 Sprint 6（裁决见 §4，Maria 已确认 2026-07-08）。
- **活动流持久化 / 完整 trace 回放 / ReAct 内层 messages 进 checkpoint**——#5 只做内存尾部渲染让用户"感知在动"；事后审计能力留给 tracing 候选线。
- **评测体系 benchmark 集**——独立 Sprint（先后关系见 §4.2）。
- **为沙箱新建 LLM API 代理/mock 服务层**——凭证缺口走 S5-01 前置识别 + sp4 已有 extra_env 注入通道解决，不另起炉灶。

## 3. 需求详述

### S5-01 凭证/外部资源需求前置识别 + 用户显式降级（#10，P0）

- **问题**：现场核实 collected_inputs=={}、无 interrupt#3、无 .secrets——sp4 交互工具全程未触发。三环失守：planning 不声明凭证需求 → coding 写不调 API 的模拟从而"不缺凭证"→ 宽松口径使绕过合法。#10 治"造假动机"（事前把资源缺口暴露给用户决策），与 S5-03 的"造假检测"（事后审计）互为攻守。
- **需求**：
  1. planning 产出**所需外部凭证/资源声明**（最小形态：一个列表，每项含名称 + 用途说明，如 "OPENAI_API_KEY：论文方法依赖真实 LLM 调用"；不做类型分级枚举）。
  2. coding 开工前比对 `secrets_store`（sp4 `.secrets` + purpose_key 机制天然复用）：缺失 → 强制经 interrupt#3（`request_user_input`）向用户索要；用户可显式选择"无凭证，降级为模拟"。
  3. 用户选择降级时，降级标记写入 state 并全链路传导（execution_result → 报告强制声明），与 S5-04 联动。
- **约束性提示**（引反馈方向，实现交架构师）：planning 产出 required_credentials 字段；coding prompt 红线补"不得以改变实验本质的方式规避资源缺失"（与 S5-02 合并落地）。
- **验收口径**：AC-S5-01 ~ 03。

### S5-02 coding 诚实红线 + simulation 声明（#9 事前，P0）

- **问题**：现场解剖证据链三环——自造任务（verifier 即关键词清单）、循环自证（verifier 答案原样抄进被评估产物，出题人=答题人）、硬编码剧本（baseline 分数写死，"复现论文趋势"是预写结局）。现状 coding 纪律未约束"实现忠于论文方法"。
- **需求**：coding prompt 增加三条红线——①禁止 verifier 答案泄漏给被评估对象；②禁止硬编码分数/结果/常量结局；③不得以改变实验本质的方式规避资源缺失。无法真实验时，必须显式产出 simulation 声明并写入 state（供 S5-03/S5-04 消费）。
- **边界**：prompt 红线是"降低发生率"手段，不承诺根除——根除靠 S5-01（消动机）+ S5-03（事后审计）三线合围。红线文案注意 R-PC4 prompt 主体冻结约束（落点交架构师）。
- **验收口径**：AC-S5-04。

### S5-03 真实性审计 + 结论降档（#9 事后，P0）

- **问题**：造假代码一路绿灯到 full_success，系统无任何审计环节。
- **需求**：在 execution 或 reporting 侧（落点交架构师，见 Q-S5-5）增加**规则型评估闭环 smell 检测**，最小规则集三类起步：①生成器读取 verifier 答案字段（答案泄漏）；②硬编码 score/结果常量；③评估结果与输入无关（常量结局）。命中任一 → 结论强制降档（禁最高档）+ 报告显著标注"模拟/未验证"。
- **边界**：规则型检测，不做 LLM-as-judge（非目标）；误报防线与命中防线同权重验收（干净代码不得降档）。审计结论进 state，判定逻辑确定性。
- **面试价值**：reward hacking 检测，eval 方向最亮叙事点；建议完成后产出 docs/reports/ HTML report。
- **验收口径**：AC-S5-05 ~ 06。

### S5-04 报告结论两级分级 + 定性目标回验 + 模拟强制声明（#7，P0）

- **问题**：11.5 秒/3 个自造任务的 smoke 模拟被表述成"✅ 复现成功"；计划定性目标（reproduction_goal 相对趋势）报告只字未回验（本次数据实际可判"部分符合"）；"缩小版 smoke 模拟"关键事实未声明。
- **需求**：
  1. 用户可见结论区分**工程复现**（代码跑通：程序正常结束、输出有效数值）与**科学复现**（趋势/指标与论文在可接受偏差内一致）两级；仅达工程复现时禁止出现"复现成功"表述，须表述为"代码跑通（工程复现），论文实验结论未验证"。
  2. 叠加**正交标注维度**（非新档位）：模拟 / 降级 / 未完整执行（来源：S5-01 降级标记、S5-02 simulation 声明、S5-03 审计命中、S5-06 对账缺口），任一命中 → 报告显著声明且不得宣告科学复现。
  3. 报告新增"计划目标回验"节：对照计划定性预期逐条给出**符合 / 不符 / 未验证**三态；存在"不符/未验证"时不得宣告完全复现成功（整体判定规则交架构师，产品只定此红线）。
  4. 现状 full_success/B 档等内部档位如何映射到两级口径，交架构师；用户可见文案走 S5-09 映射表。
- **依赖**：回验的数据基础是 S5-10（多组指标解析）；回验的对象是 S5-05（定性预期）+ 现有 reproduction_goal。
- **面试价值**：eval 判定口径设计核心叙事点。
- **验收口径**：AC-S5-07 ~ 09（09 与 S5-05 共用）。

### S5-05 expected_results 定性化（#3，P1）

- **问题**：planning prompt 要求 expected_results 引用论文指标，诱导 LLM 在论文指标缺失时凭空 mock 一套数据；现场报告 expected 列整段复述论文数字 + 混入定性 target，巨型 dict 撑爆表格。
- **需求**：
  1. planning 的 expected_results 只做**定性描述**（如"loss 应收敛""指标应接近论文报告量级""相对趋势 A>B"），prompt 明确禁止编造具体数值；论文真实 baseline 数字由 paper_analysis 提取、不在计划复述。
  2. schema 从 {metric: number} 数值映射改为定性描述形态（最小形态交架构师）。
  3. 连带：reporting 指标对比表去掉"计划 expected"列，只留论文 baseline vs 本次复现值。
- **验收口径**：AC-S5-09。

### S5-06 计划步骤执行对账 + 轮数预算联动（#11，P0）

- **问题**：计划 13 步、REACT_MAX_ROUNDS_EXECUTION=10，实际只跑 8 步后预算见顶，agent 宣布 success=True，5 步无声丢弃、报告零痕迹。>~8 步的计划结构上永远跑不完。
- **需求**：
  1. **对账**：planned vs executed 步骤对账写入 execution_result；报告展示"已完成 N/M 步 + 未执行步骤清单"。
  2. **禁静默成功**：计划步骤未全部执行时结论强制降档（与 S5-04 正交标注联动），禁止 success 无痕通过。
  3. **预算联动**：execution 轮数预算与计划步数联动（如 max_rounds = len(steps) + K 裕量，设硬上限防失控；取值交架构师，Q-S5-7）。
  4. **反模式通则**：本项落地时确立"任何 top-N/预算截断必须显式 log + state 记录"为项目通则，sp5 内至少覆盖 execution 轮数预算这一处。
- **约束性提示**：对账数据来源的可信度（agent 自报 vs 编排层确定性记账）交架构师（Q-S5-6）；产品要求对账结果不能仅依赖 agent 单方声明。
- **验收口径**：AC-S5-10 ~ 12。

### S5-07 agent 活动流最小渲染（#5，P1）

- **问题**：coding/execution 节点内 ReAct agent 跑几十轮循环，UI 全程黑盒（checkpoint 仅节点边界更新，节点内活动无通道可出）；沙箱跑代码期间 logs 是上一轮旧数据。
- **需求**（Maria 原话定边界：不用全部渲染，类似 Claude Code 方式）：
  1. 工具调用一行摘要（如"⏺ 读取 README.md""⏺ 运行 pip install…"）+ 模型输出/思考的截断预览，覆盖 coding 与 execution 两节点内活动。
  2. 执行监控页轮询渲染最近若干行活动流，复用现有轮询节奏。
  3. 事件缓存内存封顶（条数上限交架构师），不持久化、不进 checkpoint。
- **约束性提示**（引反馈方向，倾向最小方案勿过度设计）：per-thread LangChain BaseCallbackHandler 采集压缩事件、controller 经 config callbacks 传播、**不改 react_base 内部**、GraphController 上 per-thread 内存 deque——作为约束性提示交架构师验证可行性，非实现规格。
- **面试价值**：observability 一等优先级；建议完成后产出 docs/reports/ HTML report。
- **验收口径**：AC-S5-13 ~ 14。

### S5-08 UI 主路由修复 + 完成判定兜底（#4 + #6，P0）

- **问题**（#4 根因链已核实）：批准计划后 progress 页把 coding/execution/reporting 合并为占位段，无逻辑切到执行监控页；全项目唯一进 execution_monitor 的入口是 interrupt#3 强制路由，顺利执行时永远进不去；跳报告页的唯一逻辑因此不可达——复现成功用户也看不到报告。#6：report_path 为空时监控页无限假轮询。
- **需求**：
  1. current_step 进入 coding/execution → 自动切执行监控页（progress 页或统一路由层，落点交架构师）。

## 4. 两条候选线的裁决

### 4.1 「token/成本追踪 + observability」候选：**不并入 sp5**（PM 裁决，Maria 已确认 2026-07-08，Q-S5-1）

TODO 中该候选（estimate_tokens 接入、tracing 平台接入、单篇成本数字与调用链瀑布图）与 sp5 主题重合度高，但裁决**不并入**，理由：

1. **容量**：sp5 已承载 11 条反馈（11 项需求、6 项 P0），再叠一条独立工程线会挤压"假成功链"斩断的完成质量——那才是本 Sprint 的命门。
2. **耦合方向**："单篇论文复现成本"数字在 benchmark 跑批（评测体系）语境下才有统计意义，与评测体系的耦合高于与 sp5 的耦合。**建议与评测体系合成 Sprint 6**，一次跑批同时产出成功率表 + 成本表，面试叙事上也是一份更完整的 eval 报告。
3. **衔接自然**：sp5 的 S5-07 活动流事件通道客观上是后续 tracing 的地基（同为"节点内活动外化"），Sprint 6 接入 tracing 时有现成落点——但 sp5 **不为此预留任何投机性扩展点**（极简红线）。

### 4.2 「评测体系」候选：独立成 Sprint（大概率 Sprint 6），sp5 是其前置

评测体系要定义"分级成功指标（环境搭建成功/代码跑通/复现指标进容差）"——这正是 sp5 S5-04 两级结论口径的批量化应用。**顺序必须是 sp5 在前**：先把单篇判定口径做诚实（否则 benchmark 统计出的"成功率"会被假成功污染，数字不可信），Sprint 6 再基于诚实口径跑 3-5 篇 benchmark 出结果表与失败归因。
