# 全局 PRD（docs/product-design-specification.md）Sprint 5 立项审查结论

> 产出：产品经理代理，2026-07-07（随 sp5 PRD 草案 v1 一并产出）
> 状态：**已应用（2026-07-08）**。Maria 于 2026-07-08 确认 sp5 PRD Q-S5-1~4 后，R1~R13 已由主控一次性批量应用至全局 PRD（v0.2 → v0.3）；R9 附带条件项（页面 4/5 产物路径展示）随 Q-S5-2 确认一并落地。本文档留档作为 v0.3 修订依据。

总体结论：本次真跑暴露的"假成功"与全局 PRD 的"结果可信""过程透明"两条价值承诺存在直接张力——全局 PRD 需补入诚实性分级、降级透明化表述，并在功能清单/路线图反映 sp5。共 13 项修改点（12 项文本 + 1 项元数据），其中 11 项【产品决策】、2 项【事实对齐】。

## R1｜§1.3 核心价值主张·"结果可信"行｜【产品决策】

- 原文：`| 结果可信 | 自动对比复现结果与论文报告，分析差异原因，提供可靠的结论 |`
- 改为：`| 结果可信 | 自动对比复现结果与论文报告，分析差异原因；结论按"工程复现（代码跑通）/ 科学复现（趋势与指标对上）"分级如实表述，实验为模拟或缩减规模时强制声明，不以宽松口径冒充复现成功 |`
- 理由：真跑证明"提供可靠的结论"承诺在宽松口径下不成立；价值主张须显式纳入诚实性分级，否则承诺与 sp5 判定口径脱节。

## R2｜§1.3 核心价值主张·"过程透明"行｜【产品决策】

- 原文：`| 过程透明 | 完整的复现规划供用户审核和调整，而非黑盒操作 |`
- 改为：`| 过程透明 | 完整的复现规划供用户审核和调整；执行阶段实时呈现 agent 活动摘要，计划步骤完成情况（N/M 对账）与任何降级、截断、模拟均显式可见，而非黑盒操作 |`
- 理由：现行表述只覆盖规划阶段透明；#5（执行黑盒）/#11（静默截断）证明执行阶段透明须成为承诺的一部分。

## R3｜§3.2 步骤 4：复现规划与用户审核·输出行｜【产品决策】

- 原文：`| **输出** | 完整的复现计划文档（包括环境配置、执行步骤、预期耗时、资源需求） |`
- 改为：`| **输出** | 完整的复现计划文档（包括环境配置、执行步骤、预期耗时、资源需求、所需外部凭证/资源清单） |`
- 理由：sp5 S5-01 凭证前置识别要求计划显式声明外部凭证/资源需求，步骤定义须同步。

## R4｜§4.3.1 复现计划内容·计划结构树（同一 code block 内两处）｜【产品决策】

- 原文（其一）：

```
├── 2. 环境需求
│   ├── 硬件要求（GPU型号/数量、内存、磁盘空间）
│   ├── 软件依赖（框架版本、关键库）
│   └── 预估运行时间
```

- 改为：

```
├── 2. 环境需求
│   ├── 硬件要求（GPU型号/数量、内存、磁盘空间）
│   ├── 软件依赖（框架版本、关键库）
│   ├── 所需外部凭证/资源（如 LLM API key、私有仓库 token；缺失时的处置见 §5.3）
│   └── 预估运行时间
```

- 原文（其二）：

```
└── 6. 预期结果
    ├── 论文报告的指标值
    └── 可接受的偏差范围（如有经验参考）
```

- 改为：

```
└── 6. 预期结果（定性）
    ├── 应观察到的趋势与量级（定性描述，禁止编造论文未报告的具体数值）
    └── 判定复现符合预期的定性依据（供报告逐条回验）
```

- 理由：其一对应 S5-01 前置识别；其二对应 #3 现场实锤——"论文报告的指标值"写进计划正是诱导 LLM 编造数字的源头表述，改为定性并成为 S5-04 报告回验的对象。

## R5｜§4.5.2 结果验证·核心标准 bullet｜【产品决策】

- 原文：`- **核心标准**：实验能跑出结果（程序正常结束，输出有效数值）`
- 改为：

```
- **核心标准（Sprint 5 修订为两级结论）**：
  - **工程复现**：实验能跑出结果（程序正常结束，输出有效数值）
  - **科学复现**：复现结果与论文报告的趋势/指标在可接受偏差内一致
  仅达到工程复现时，不得表述为"复现成功"；报告须对照复现计划中的定性预期逐条给出"符合/不符/未验证"
- **真实性要求（Sprint 5 新增）**：复现代码须忠于论文方法。系统对"评估自证闭环（出题人=答题人）、硬编码分数/结果"等造假模式进行审计，命中则结论降档并在报告显著标注；实验为模拟或缩减规模时必须在报告中强制声明
```

- 理由：现行"核心标准=能跑出结果"正是 #7 宽松口径的产品源头（B 档"代码跑通"被合法表述为复现成功的依据就在这一句），是全文最必须改的一处。

## R6｜§4.5.3 错误处理策略·重试预算段末句｜【产品决策】

- 原文：`预算对用户不透明、不可调——这是工程保护措施，不是产品功能。预算接近耗尽时给出 WARNING 通知；最终报告中记录总修复尝试次数。`
- 改为：`预算取值对用户不透明、不可调——这是工程保护措施，不是产品功能。预算接近耗尽时给出 WARNING 通知；最终报告中记录总修复尝试次数。执行阶段的轮次预算与计划步数联动分配（设硬上限防失控）；当预算耗尽导致计划步骤未全部执行时，系统必须显式对账（报告展示"已完成 N/M 步"及未执行步骤清单）并使结论降档，禁止静默宣告成功（Sprint 5，详见 docs/sprint5/prd.md）。`
- 理由：#11 实锤"预算不透明"被执行成了"预算截断后果也不透明"。改"预算"为"预算取值"厘清边界：取值不暴露，但截断后果必须透明——化解与 R2 透明承诺的张力。

## R7｜§4.5.3 错误处理策略·错误信息展示规范措辞行｜【产品决策】

- 原文：`措辞规范：不使用"系统异常""未知错误"；技术术语可用但不堆砌；所有错误信息必须给出"下一步"。`
- 改为：`措辞规范：不使用"系统异常""未知错误"；技术术语可用但不堆砌；系统内部枚举值与自创术语（如 from_scratch、full_success、error_category 取值、内部节点名）不得裸露给用户，UI 展示与 LLM 生成的用户可见文本均使用通俗中文表述，机器可读字段（JSON 内枚举值）不受此限（Sprint 5）；所有错误信息必须给出"下一步"。`
- 理由：#1/#2 术语治理需在全局措辞规范中固化为通则，而非仅 sp5 一次性清理。

## R8｜§4.6.1 报告核心内容｜【产品决策】

- 原文：

```
每份复现报告必须包含以下三部分：

1. **复现结果**：完整的实验结果数据，包括所有评估指标的数值
2. **与原文对比**：以表格形式对比复现结果与论文原始数据
3. **不一致原因分析**：当结果存在差异时，给出可能的原因分析
```

- 改为：

```
每份复现报告必须包含以下四部分：

1. **复现结果**：完整的实验结果数据，包括所有评估指标的数值
2. **与原文对比**：以表格形式对比复现结果与论文原始数据
3. **不一致原因分析**：当结果存在差异时，给出可能的原因分析
4. **结论分级与诚实声明（Sprint 5 新增）**：明示结论档位（工程复现 / 科学复现，见 §4.5.2）；对照复现计划的定性预期逐条给出"符合/不符/未验证"；实验为模拟、缩减规模或存在未执行的计划步骤（N/M 步对账）时强制显著声明
```

- 理由：报告是假成功链的最后一环出口，核心内容定义须把诚实声明升为硬性构成部分。

## R10｜§5.3 执行过程中的信息索取交互·触发场景 bullet｜【产品决策】

- 原文：`- **触发场景**：coding / execution agent 在执行中缺失任何推进所需的关键信息（凭证 / 数据路径 / 参数 / 需人工决策的选择），最典型的是凭证缺失。`
- 改为：`- **触发场景**：coding / execution agent 在执行中缺失任何推进所需的关键信息（凭证 / 数据路径 / 参数 / 需人工决策的选择），最典型的是凭证缺失。Sprint 5 起新增**前置识别**：复现计划中声明所需外部凭证/资源，编码开工前主动比对已存凭证，缺失即发起本交互——不等执行中途报错才发现；用户可显式选择"无凭证降级为模拟"，该降级将全链路标注并在报告中强制声明（agent 不得自行决定降级）。`
- 理由：真跑证明"执行中缺失才触发"的被动模型可被 agent 用降级实现绕过（#10：写不调 API 的模拟就"不缺凭证"），交互触发模型须升级为"前置比对 + 被动兜底"。
